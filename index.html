1. One-Sentence Machine Definition

Claude Code executes a stateful, permission-gated, tool-calling agent loop that ingests a natural-language prompt, iteratively dispatches LLM API calls and tool invocations until no tool-use blocks remain, and persists the full transcript as append-only JSONL while maintaining a multi-level context window compaction stack to keep token usage within model limits.

2. Why This Part Exists

The system exists to solve a specific reliability problem: raw LLM completions cannot safely write files, run shell commands, or perform multi-step operations on a real filesystem without (a) a deterministic dispatch layer that routes each model-requested tool call to the correct implementation, (b) a permission gate that enforces human-in-the-loop consent before destructive actions, and (c) a memory architecture that survives context window overflow across long sessions. Without all three, a pure chat interface would either silently fail on complex tasks or destructively modify the filesystem without user awareness.

Reel Creator faces the identical problem domain shifted to content: raw LLM completions cannot safely publish to TikTok/Instagram, call asset generation APIs, or manage a multi-platform publishing pipeline without the same three components. The architectural motive transfers exactly; the tools change entirely.

5. Happy-Path Surgical Walkthrough

Scenario: User runs claude "fix the bug in auth.ts" from the terminal.

Step 1 — Binary reads argv: src/entrypoints/cli.tsx:main() receives process.argv = ["node", "claude", "fix the bug in auth.ts"]. No fast-path flags detected. Delegates to src/main.tsx.

Step 2 — Commander parses flags: src/main.tsx parses "fix the bug in auth.ts" as the [prompt] positional argument. Activates headless print path. Constructs headlessStore via createStore(headlessInitialState).

Step 3 — QueryEngine initializes: Constructor sets this.mutableMessages = [], creates AbortController, initializes readFileState.

Step 4 — submitMessage() assembles context: Calls fetchSystemPromptParts() which loads base system prompt, CLAUDE.md hierarchy, and git status.

Step 5 — User input processes: processUserInput() receives the text, finds no slash command, creates a UserMessage, pushes to mutableMessages, records transcript.

Step 6 — query() delegates to queryLoop(): Passes assembled messages, system prompt, tools list, and dependency injection object.

Step 7 — while(true) loop begins: Compaction checks run. Calls deps.callModel() which resolves to queryModelWithStreaming.

Step 8 — LLM streams response: Model responds with a tool_use block requesting FileReadTool for auth.ts.

Step 9 — Tools dispatch: runTools() partitions blocks. FileReadTool is read-only, runs concurrently.

Step 10 — Permission gate and tool execution: checkPermissionsAndCallTool() validates input with Zod schema. Read-only operations resolve to allow. Calls tool.call().

Step 11 — Tool result feeds back: FileReadTool reads auth.ts contents. Result yields as UserMessage with tool_result block. Loop continues.

Step 12 — Second LLM call: Loop iteration 2 includes file contents. Model returns FileEditTool tool_use block.

Step 13 — Write permission check: Write operation requires confirmation. User sees Accept/Reject UI. User accepts.

Step 14 — Edit applies and loop terminates: FileEditTool.call() applies patch. Loop iteration 3: model returns zero tool_use blocks. Stop condition met. Transcript finalized.

6. Dispatch Map

LLM → tool dispatch: Model emits tool_use blocks with a name field. runTools() receives blocks, partitions by read-only classification, dispatches via runToolUse(). findToolByName() locates the tool object.

Tool registry: src/tools.ts:getAllBaseTools() — hardcoded list: AgentTool, BashTool, FileEditTool, FileReadTool, FileWriteTool, GlobTool, GrepTool, NotebookEditTool, TodoWriteTool, SkillTool, ToolSearchTool, and others. MCP tools dynamically instantiate as MCPTool objects.

Slash command dispatch: processUserInput() calls parseSlashCommand() to detect leading /, then findCommand() to locate matching Command object. Commands with call() execute immediately. Commands returning prompt strings forward to model.

Subagent spawning: runAgent() creates isolated execution context via createSubagentContext(). Clones parent's file state cache, creates child AbortController, assigns new agentId, calls query() recursively.

MCP tool loading: connectToServer() creates transport, calls client.listTools(), instantiates MCPTool objects.

Skill loading: loadSkillsDir.ts scans skill directories. Each .md file with valid YAML frontmatter becomes a PromptCommand. When model calls SkillTool, it runs as forked agent sub-query.

  1. String → Commander ParsedArgs: Commander.js parses argv into structured arguments.
  2. ParsedArgs → UserMessage: processUserInput() wraps string in typed UserMessage object.
  3. UserMessage → API Request: submitMessage() assembles full request with system prompt, messages, tools, model config.
  4. API Request → Streaming Events: SDK streams events, assembled into AssistantMessage.

  9. Final AssistantMessage → JSONL: Each message appends to session transcript.

10. Important Branch Points

  • Permission mode branch: Evaluates deny rules first, then allow rules, falls through to mode-based default.
  • Interactive vs. headless: --print flag or positional prompt routes to headless path.
  • Tool batching: Partitions into read-only (concurrent) vs. non-read-only (serial).
  • Context window overflow: Triggers compaction stack: Snip → Microcompact → Context Collapse → Autocompact → Reactive Compact.
  • Subagent vs. inline: AgentTool and SkillTool spawn isolated recursion.
  • Fallback model: FallbackTriggeredError switches to configured fallback.

11. Failure and Recovery Logic

  • Permission denial: Returns tool_result with is_error: true.
  • Input validation failure: Zod parse fails → InputValidationError → error result to model.
  • Runtime tool error: Exception caught, wrapped as error result, fed back to model.
  • LLM API rate limiting: Exponential backoff with jitter on 429 responses.
  • Fallback model: Switches mainLoopModel to fallback, retries current turn.
  • Max output tokens recovery: Up to 3 retries by appending continuation meta-message.
  • Context window overflow stack: Five mechanisms at progressively higher severity.
  • Human-in-the-loop: Queues confirmation dialog for uncertain permissions.

13. Architectural Compression

Permission model: Four modes with deny-before-allow evaluation across seven source layers. Conservative denial: any deny rule blocks.

Memory architecture: Four independent time horizons:

  • CLAUDE.md: Static instructions, 4-layer hierarchy, survives restarts.
  • MEMORY.md: Model-writable persistent memory, survives sessions.
  • Session transcript: Append-only JSONL, resumable.
  • Invoked skills cache: Tracks skills called in current session.

Skill system: Markdown files with YAML frontmatter. Skills compose with subagents: SkillTool forks new query() call.

14. Design Intent Inference

  • Loop-as-generator pattern: Optimized for streaming UI updates, makes loop pausable for --resume.
  • Dependency injection: Designed for testability and swapping LLM backend.
  • YAML frontmatter skills: Designed for non-engineer authorship, documentation-first.
  • Four-source permission hierarchy: Intent to operate in enterprise environments where IT policy overrides user preferences.
  • DeepImmutable AppState: UI components forbidden from mutating state directly.
  • "DO NOT ADD MORE STATE HERE" comment: Architectural confession about bootstrap singleton growth.
  • Five-level compaction stack: Grew incrementally as users encountered longer sessions, not designed upfront.

1. 单句机器定义

Claude Code 执行一个有状态的、权限门控的、工具调用代理循环，它接收自然语言提示，迭代地分发 LLM API 调用和工具调用，直到不再有 tool-use 块为止，并将完整记录持久化为仅追加的 JSONL，同时维护多级上下文窗口压缩栈以将令牌使用量保持在模型限制内。

2. 存在原因

该系统的存在是为了解决一个特定的可靠性问题：原始 LLM 补全无法安全地写入文件、运行 shell 命令或在真实文件系统上执行多步操作，除非具备：(a) 确定性调度层，将每个模型请求的工具调用路由到正确的实现；(b) 权限门，在破坏性操作之前强制执行人机协同确认；(c) 能够在长会话中跨越上下文窗口溢出而存活的内存架构。如果缺少这三者中的任何一个，纯聊天界面要么在复杂任务上静默失败，要么在用户不知情的情况下破坏性地修改文件系统。

Reel Creator 面临着相同的问题域，只是转移到了内容领域：原始 LLM 补全无法安全地发布到 TikTok/Instagram、调用资产生成 API 或管理多平台发布管道，除非具备相同的三个组件。架构动机完全转移；工具完全改变。

6. 调度映射

LLM → 工具调度： 模型发出带有 name 字段的 tool_use 块。runTools() 接收块，按只读分类分区，通过 runToolUse() 调度。findToolByName() 定位工具对象。

工具注册表： src/tools.ts:getAllBaseTools() — 硬编码列表：AgentTool、BashTool、FileEditTool、FileReadTool 等。MCP 工具动态实例化为 MCPTool 对象。

斜杠命令调度： processUserInput() 调用 parseSlashCommand() 检测前导 /，然后 findCommand() 定位匹配的 Command 对象。

子代理生成： runAgent() 通过 createSubagentContext() 创建隔离的执行上下文。克隆父级的文件状态缓存，创建子 AbortController，递归调用 query()。

MCP 工具加载： connectToServer() 创建传输，调用 client.listTools()，实例化 MCPTool 对象。

技能加载： loadSkillsDir.ts 扫描技能目录。每个带有有效 YAML 前置内容的 .md 文件成为 PromptCommand。当模型调用 SkillTool 时，它作为分叉代理子查询运行。

7. 状态模型

AppState： 面向 UI 和会话级状态的主要响应式存储。字段包括：settings、mainLoopModel、toolPermissionContext、mcp、plugins、tasks、fileHistory 以及所有 UI 显示字段。

查询循环状态： queryLoop() 内部的局部可变状态 — messages[]、toolUseContext、autoCompactTracking、turnCount。仅在一次 queryLoop() 调用的生命周期内存在。

引导状态： 模块级全局变量 — sessionId、totalCostUSD、modelUsage、isInteractive、registeredHooks。第 32 行的注释："不要在这里添加更多状态。"

对话历史： 双重表示 — 在 QueryEngine 中作为 mutableMessages[] 的内存表示，以及作为仅追加 JSONL 的磁盘表示。

工具权限状态： mode（default/plan/bypassPermissions/auto）、alwaysAllowRules、alwaysDenyRules、alwaysAskRules 按来源键控。

8. 数据转换链

  1. 字符串 → Commander ParsedArgs： Commander.js 将 argv 解析为结构化参数。
  2. ParsedArgs → UserMessage： processUserInput() 将字符串包装在类型化的 UserMessage 对象中。
  3. UserMessage → API 请求： submitMessage() 组装完整请求，包含系统提示、消息、工具、模型配置。
  4. API 请求 → 流式事件： SDK 流式传输事件，组装成 AssistantMessage。
  5. AssistantMessage → ToolUseBlock[]： 循环收集 tool_use 内容块。
  6. ToolUseBlock → Zod 验证输入： 每个块的输入通过工具的 Zod 模式。
  7. 验证输入 → 工具结果： 工具执行，返回 tool_result。
  8. 工具结果 → UserMessage： 包装为 UserMessage，追加到 messages[]。
  9. 最终 AssistantMessage → JSONL： 每条消息追加到会话记录。

9. 副作用映射

文件系统写入： 会话记录、文件编辑、配置保存、内存写入、撤销快照。

LLM API 调用： 每次循环迭代的主要模型调用，加上压缩摘要、内存提取、代理摘要、提示建议。

子进程生成： 带有 sandbox-exec 包装器的 BashTool、MCP stdio 服务器。

MCP 网络调用： client.callTool()、client.listTools()、client.readResource()。

持久化路径： ~/.claude/projects/、~/.claude.json、.claude/settings.json、~/.claude/memory/。

10. 重要分支点

  • 权限模式分支： 首先评估拒绝规则，然后是允许规则，最后回退到基于模式的默认值。
  • 交互式 vs 无头： --print 标志或位置提示路由到无头路径。
  • 工具批处理： 分区为只读（并发）vs 非只读（串行）。
  • 上下文窗口溢出： 触发压缩栈：Snip → Microcompact → Context Collapse → Autocompact → Reactive Compact。
  • 子代理 vs 内联： AgentTool 和 SkillTool 生成隔离递归。
  • 回退模型： FallbackTriggeredError 切换到配置的回退。

11. 故障和恢复逻辑

  • 权限拒绝： 返回带有 is_error: true 的 tool_result。
  • 输入验证失败： Zod 解析失败 → InputValidationError → 错误结果返回给模型。
  • 运行时工具错误： 异常被捕获，包装为错误结果，反馈给模型。
  • LLM API 速率限制： 对 429 响应进行指数退避和抖动。
  • 回退模型： 将 mainLoopModel 切换到回退，重试当前轮次。
  • 最大输出令牌恢复： 通过追加延续元消息最多重试 3 次。
  • 上下文窗口溢出栈： 五种机制，严重程度逐渐升高。
  • 人机协同： 为不确定的权限排队确认对话框。

14. 设计意图推断

  • 循环即生成器模式： 针对流式 UI 更新进行优化，使循环可暂停以支持 --resume。
  • 依赖注入： 为可测试性和交换 LLM 后端而设计。
  • YAML 前置内容技能： 为非工程师编写而设计，文档优先。
  • 四源权限层次结构： 意图在企业环境中运行，IT 策略覆盖用户偏好。

  • "不要在这里添加更多状态"注释： 关于引导单例增长的架构坦白。
  • 五级压缩栈： 随着用户遇到更长的会话而逐步增长，而非预先设计。

18. 心智模型包

  • queryLoop() 迭代：调用 LLM、收集 tool_use 块、执行工具、反馈结果、当剩余零个块时停止。
  • 每个工具调用都通过 hasPermissionsToUseTool()；拒绝规则在七个源层的允许规则之前评估。
  • 响应式存储强制执行硬边界：代理循环修改状态，UI 只读取。
  • CLAUDE.md 注入四层静态上下文；MEMORY.md 提供模型可写的持久内存。
  • 技能是带有 YAML 前置内容的 Markdown 文件；每个都生成隔离的分叉 query() 调用。
  • 会话状态存在于三个地方：内存中的 mutableMessages[]、响应式 AppState、磁盘上的仅追加 JSONL。
  • 五级压缩栈的存在是因为上下文溢出是逐步显现的。
  • src/main.tsx 是一个 4,683 行的上帝文件；其他一切都相对干净且可以外科式替换。
